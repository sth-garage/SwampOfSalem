Issue 1 — "An unhandled error has occurred. Reload 🗙" · No squares on page

User
I'm seeing an "An unhandled error has occurred. Reload 🗙" but there are no errors in the console. There are no squares on the page.
GitHub Copilot

Two root causes identified — both CSS-only, no runtime .NET exceptions.

Bug 1 — Error banner always visible

Blazor's #blazor-error-ui div is always present in the DOM. Blazor JS only sets display:block on it when a real error occurs — it relies on CSS to keep it hidden at startup. app.css (which normally contains #blazor-error-ui { display: none }) was never linked in index.html; only people-watcher.css was linked.

Bug 2 — Person squares invisible (zero-height world)

The flex layout chain was broken: body (flex column, 100% height) → #app → #world { flex:1 }. #app had no flex styles, so #world collapsed to clientHeight = 0. stageBounds() returned { W: -80, H: -80 } and overflow:hidden clipped every person element.

Fix — added to people-watcher.css

/* Blazor app root — must be a flex column so #world can flex-grow */
#app {
    display: flex;
    flex-direction: column;
    flex: 1;
    min-height: 0;
}

/* Hide the error banner unless Blazor surfaces a real error */
#blazor-error-ui { display: none; }

Turn 12 — Full Relationship System

User
Keep track of the relationship between each person to another person. Some like each other, some hate each other. Their opinions may change based on conversations or how one person treats another person's friend or enemy. Some people lie and pretend to like people they really don't.
GitHub Copilot

Implemented a full pairwise relationship system in people-watcher.js. All changes were additions/ modifications to the existing engine — no files were created or removed.

New constants

  • LIAR_CHANCE — per-personality probability of being a liar (extroverts highest at 25 %, grumpy lowest at 5 %)
  • COMPAT — 6×6 personality compatibility matrix; determines how much two people's relationship shifts after chatting (e.g. cheerful + extrovert bond quickly; grumpy + energetic clash)
  • RELATION_THOUGHTS — five tiers of internal thought snippets (love / like / neutral / dislike / hate) that surface in the thought bubble after a conversation ends

New state per person

  • relations[id] — true feeling toward every other person, −100 → +100
  • perceivedRelations[id] — what they show; liars show false warmth when feeling < −20
  • liar: boolean — set randomly on spawn according to LIAR_CHANCE

New functions

  • initRelations() — called once after all people are created; seeds every pair with a random −100→+100 base feeling; liars get a desynced perceivedRelations
  • driftRelations(a, b) — called at the end of every conversation:
    • Applies compatibility bonus/penalty from COMPAT
    • Third-party alignment: if B likes person C and A also likes C, A and B bond more; if they disagree about C they drift apart
    • Clamps both sides to −100/+100
    • Re-syncs liars' perceived values
  • relationColor(avg) — returns green / yellow / red stroke colour for talk lines
  • relationEmoji(val) — returns ❤️/😊/😐/😒/😡 for tooltip display
  • pickRelationThought(val) — picks a thought snippet matching the tier of the feeling

Behaviour changes

  • Talk targeting — people avoid initiating conversations with anyone they hate (< −60); among willing candidates they prefer the most-liked person via a weighted comparison
  • Walk-toward — when seeking someone to talk to, they walk toward their most-liked available person instead of the most socially needy
  • Thought bubbles — after each conversation ends, both participants get a relation-aware thought ("I could talk to them forever." / "I absolutely can't stand them.") lasting 4–8 ticks
  • Talk lines — colour reflects the true average feeling between the pair: green (avg ≥ 30), yellow (−30 to 30), red (avg < −30)
  • Tooltip — now shows a full relationship roster for the hovered person: true value, emoji tier, and a (lies) tag on any row where their shown feeling differs significantly from the truth; liars get a 🎭 badge next to their name

Problem: everyone talks at once

During debate, all 10 people received an accusation message simultaneously in triggerDebate(). During day conversations, both talkers refreshed their speech bubble every single tick. This made speech feel robotic and chaotic.

Solution: staggered speech system

  • New speakCooldown property on every person — counts ticks until they can show a new chat bubble.
  • New constants: MAX_DEBATE_SPEAKERS = 2 (max simultaneous speakers during debate), DEBATE_SPEAK_COOLDOWN = [2, 4] (ticks between debate utterances).
  • Debate phase: triggerDebate() shuffles the living people. The first MAX_DEBATE_SPEAKERS speak immediately; everyone else gets a random initial delay of 1–4 ticks. On each tick, only people whose cooldown has expired and fewer than MAX_DEBATE_SPEAKERS are currently speaking get to talk. After speaking, they enter a 2–4 tick cooldown. Messages are cleared 1 tick before the next utterance for a visible gap.
  • Day conversations: When two people start talking, the initiator speaks immediately but gets a 1–2 tick cooldown; the partner gets a 2–3 tick cooldown, so they reply slightly later. Subsequent messages appear only when cooldown expires (every 1–3 ticks instead of every tick).
  • Murderer bluffs: Also gated by speakCooldown, with a 2–3 tick gap between bluff lines.

New projects

ProjectTypePurpose
GatorGazing.SharedClass libraryEnums (GamePhase, Personality, DialogType), domain models (Alligator, GameState), and DTOs (AgentDialogRequest/Response, AgentVoteRequest/Response, AgentMemoryRequest)
GatorGazing.AppLogicClass libraryPure game-rule services: PhaseManager, MurderService, VoteService (clockwise ordering, tallying, execution), RelationshipService
GatorGazing.SKClass librarySemantic Kernel agent layer: GatorAgentService (one ChatCompletionAgent per alligator with per-agent ChatHistory memory), SwampPlugin (tool functions for relationships, suspicion, memories), PersonalityPrompts (personality-specific system prompts with murderer/liar overrides)

Key design: per-agent memory

  • Each alligator agent maintains its own ChatHistory — every dialog, observation, and vote is appended so the LLM sees the full personal timeline when generating responses.
  • AddMemory(alligatorId, memory) inserts user-role entries so the agent “remembers” observed events (e.g. “Fang voted for Chomps on Day 2”).
  • The SwampPlugin exposes GetRecentMemories, GetRelationship, GetSuspicion, and GetAlligatorInfo as Semantic Kernel tool functions the agent can call during inference.

Debate & vote architecture

  • Debate: RunDebateRoundAsync(alligatorIds, topic) generates one round of simultaneous discussion messages from all agents.
  • Vote: GetVoteAsync(alligatorId, candidateIds, debateSummary) asks each agent individually who to vote for, producing a clockwise sequential vote.
